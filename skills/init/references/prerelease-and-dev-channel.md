# pre-release 수식어와 dev 채널

이 문서는 정식 버전이 아닌 상태를 표시하는 수식어를 어떻게 다뤄야 하는지, 그리고 dev 서버처럼 계속 재배포되는 환경에서 버전을 어떻게 취급해야 하는지를 다룬다. `version-schemes.md`가 체계 자체의 규칙을 다룬다면, 이 문서는 그 체계 위에서 수식어가 실제로 어떻게 운용되는지에 집중한다.

버전 문자열 뒤에 붙는 "수식어"는 겉보기엔 비슷해 보여도 저장소가 그 문자열을 다루는 시맨틱이 완전히 다른 두 종류로 나뉜다. 이 둘을 혼동하면 안 된다.

- **불변 카운터형**: `-alpha.1`, `-M1`, `-RC.2`처럼 한 번 발행되면 그 문자열이 다시는 바뀌지 않고, 다음 단계로 넘어갈 때는 카운터를 증가시켜 새 문자열을 만든다. SemVer의 pre-release 시맨틱이 여기에 해당하며, 소비자 입장에서 이 문자열은 "이 시점의 특정 스냅샷"을 가리키는 불변 참조로 취급된다.
- **가변 덮어쓰기형**: `-SNAPSHOT`(Maven 계열)처럼 같은 버전 문자열을 계속 다시 발행하며 덮어쓴다. 저장소에 새 아티팩트가 올라가도 버전 문자열 자체는 그대로이고, 소비자도 이 문자열을 "고정된 특정 빌드"가 아니라 "항상 최신 개발 상태를 가리키는 가변 별칭"으로 취급한다.

(superrelease에서 불변 카운터형 릴리스 플로우는 지원된다 — 발행은 `next-version.py --prerelease <qualifier>`(카운터 자동 증가), 정식 승격은 `--release`가 담당한다.)

두 모델을 나란히 비교하면 차이가 더 분명해진다.

| | 불변 카운터형 | 가변 덮어쓰기형 |
|---|---|---|
| 예시 | `-alpha.1`, `-rc.2` | `-SNAPSHOT` |
| 문자열 재사용 | 안 함 — 다음 단계는 항상 새 문자열 | 함 — 같은 문자열을 계속 덮어씀 |
| 소비자 관점 | 특정 시점의 스냅샷 | 항상 최신 개발 상태 |
| 전형적 쓰임 | 정식 릴리스 전 rc 단계 | dev 채널, 진행 중인 개발 버전 |

두 모델 모두 "아직 정식 릴리스가 아니다"라는 신호를 준다는 점은 같지만, 하나는 불변 스냅샷이고 다른 하나는 가변 별칭이라는 점에서 저장소 시맨틱이 정반대다.

이 구분을 소홀히 하면 실제로 문제가 생긴다. 예를 들어 dev 서버가 `1.3.0-SNAPSHOT`을 계속 재배포하는 동안 어느 시점을 두고 "지금 1.3.0-SNAPSHOT이 떠 있다"고만 기록해두면, 그 순간 실제로 어떤 커밋이 반영된 빌드였는지는 문자열만 봐서는 절대 알 수 없다 — 같은 문자열 아래 서로 다른 다섯 번의 배포가 있었을 수 있기 때문이다.

어느 쪽 모델을 쓰는지에 따라 재발행 가능 여부와 이후 태깅 방식이 달라지므로 항상 구분해서 다뤄야 한다.

## 웹 앱의 dev 서버 배포 (핵심 시나리오)

dev 서버에만 배포되는 빌드는 버전을 bump하지 않는다.

파일 버전은 `{다음 목표 버전}-{가변 수식어}`(예: `1.3.0-SNAPSHOT`) 형태로 유지하고, dev 서버로 나가는 배포는 매번 이 동일한 문자열을 덮어쓴다. 배포할 때마다 버전을 올려버리면 아직 정식으로 나가지도 않은 내부 빌드들이 버전 번호만 계속 소모하게 되므로, 목표 버전 자체는 고정해두고 그 뒤에만 가변 수식어를 붙이는 것이다.

문제는 가변 문자열만으로는 "지금 dev에 정확히 어떤 코드가 떠 있는지" 추적할 수 없다는 점이다. 배포 다섯 번을 해도 문자열은 매번 똑같이 `1.3.0-SNAPSHOT`이기 때문이다.

그래서 best practice는 **불변 식별자를 반드시 함께 병기**하는 것이다.

- **Spring**: `springBoot { buildInfo() }`로 빌드 정보를 기록하고 git.properties(커밋 sha 포함)를 함께 두면 `/actuator/info` 엔드포인트에서 지금 떠 있는 커밋을 확인할 수 있다.
- **Docker**: 가변 채널 태그(`dev`)와 불변 태그(`sha-{shortSha}` 또는 `{version}-{shortSha}`)를 함께 push한다. `dev` 태그는 계속 최신을 가리키지만, sha가 포함된 태그는 그 시점의 이미지를 고정해서 가리킨다.
- **npm 계열**: 필요하면 `1.3.0-dev.{sha}`처럼 커밋 sha를 아예 버전 문자열 안에 포함하는 옵션도 쓴다.

이 불변 식별자는 장애 대응 상황에서 특히 중요하다. dev 환경에서 재현되는 문제를 조사할 때, 정확히 어떤 커밋이 배포돼 있었는지 sha로 확인할 수 있어야 원인이 된 커밋 범위를 좁혀갈 수 있다.

흐름을 worked example로 보면 이렇다. 개발 중인 목표 버전이 1.3.0이라고 하자.

1. 첫 번째 dev 배포: Docker 이미지를 `dev`와 `sha-a1b2c3d`로 push. 파일 버전은 `1.3.0-SNAPSHOT`.
2. 두 번째 dev 배포: 다시 `dev`를 덮어쓰면서 `sha-e4f5g6h`라는 새 불변 태그를 추가로 push. 파일 버전은 여전히 `1.3.0-SNAPSHOT`.
3. 정식 릴리스: 파일 버전이 `1.3.0`으로 bump되고 태그가 찍힌다.

이 과정에서 `1.3.0-SNAPSHOT`이라는 표시는 한 번도 바뀌지 않지만, `sha-*` 태그를 보면 그 시점에 정확히 어떤 커밋이 배포되어 있었는지 구분할 수 있다.

불변 카운터형을 실제 릴리스 흐름에 쓰는 경우도 있다. 예를 들어 정식 릴리스 전에 `1.4.0-rc.1`, `1.4.0-rc.2`를 순서대로 발행하고 문제가 없으면 최종적으로 `1.4.0`을 릴리스하는 방식이다.

이 흐름은 각 rc가 GitHub Release의 prerelease 표시와 자연스럽게 연결된다 — superrelease의 release 스킬은 counter 스타일 scope의 pre-release 버전에 `gh release create --prerelease` 플래그를 붙이고, 승격 릴리스에는 붙이지 않는다.

요약하면 이렇다 — **사람용 라인 표시는 가변 수식어, 추적은 불변 sha. 정식 릴리스만 bump + 태그.**

## post-release 정책

릴리스 직후 다음 개발 버전으로 어떻게 돌아가는지는 레포 성격에 따라 다르다.

- **JVM 라이브러리**: 릴리스 직후 다음 개발 버전을 `-SNAPSHOT`으로 bump하는 커밋을 남긴다. Maven·Spring 생태계에서 오래 정착된 관례로, 예를 들어 1.3.0을 릴리스하자마자 그 직후 커밋으로 1.4.0-SNAPSHOT을 만들어 다음 개발 사이클을 시작한다.
- **앱/서비스**: 기본은 post-release bump가 없어 1.3.0 그대로 유지된다. 다만 dev 채널을 SNAPSHOT 방식으로 운영하고 있었다면, 릴리스 직후 `{next}-SNAPSHOT`(1.4.0-SNAPSHOT)으로 복귀하는 편이 자연스럽다.

dev 채널이 표시하는 "다음 목표 버전"과 실제 파일 버전이 앞뒤가 맞아야 하므로, 이는 사실상 JVM 라이브러리와 같은 관례를 앱에도 그대로 적용하는 셈이다.

init은 스캔으로 파악한 레포 성격에 맞는 기본값을 먼저 제안하고, 사용자에게는 그 제안에 대한 확인만 받는다 — 처음부터 백지 상태로 자유 입력을 요구하지 않는다.

JVM 라이브러리가 이 관례를 쓰는 이유는 Maven 생태계의 실용적 필요 때문이기도 하다. 사내 Maven 저장소에 `-SNAPSHOT` 아티팩트를 계속 배포해두면, 이 라이브러리에 의존하는 다른 팀이 정식 릴리스를 기다리지 않고도 `1.4.0-SNAPSHOT`을 그대로 참조해 최신 개발 버전을 미리 검증해볼 수 있다.

## config 매핑

- `preRelease.style`: `none | mutable | counter` — 수식어를 아예 안 쓰는지, 가변 덮어쓰기형인지, 불변 카운터형인지를 나타낸다.
- `devChannel`: `qualifier`(가변 수식어 문자열, 예: `SNAPSHOT`)와 `immutableId`(불변 식별자 방식 목록, 예: `spring-build-info`, `docker-sha-tag`)로 구성된다.
- `postRelease.bump`: `next-snapshot | none` — 위 정책 중 어느 쪽을 자동 적용할지를 나타낸다.

**M1에서 `devChannel`은 값을 기록하고 관련 스니펫을 안내하는 역할만 한다** — dev 배포 자체를 자동화하는 것은 CI의 몫이므로 스킬이 대신 실행하지 않는다.

`immutableId` 목록의 각 값은 위에서 소개한 방식과 대응한다 — `spring-build-info`는 Spring의 `buildInfo()` + git.properties 조합을, `docker-sha-tag`는 Docker의 `sha-{shortSha}` 병기 방식을 가리킨다.

이 세 필드는 서로 맞물려 동작한다. 예를 들어 `devChannel.qualifier`가 `SNAPSHOT`이고 `postRelease.bump`가 `next-snapshot`이면, 릴리스 직후 되돌아가는 개발 버전 문자열과 dev 채널이 계속 덮어쓰는 문자열이 같아진다.

별개의 두 정책이 아니라 한 사이클로 맞물려 돌아가는 구조라는 뜻이다.
