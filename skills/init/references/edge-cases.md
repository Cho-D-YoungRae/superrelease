# 엣지 케이스

이 문서는 스캔·질문·릴리스 과정에서 마주치는 예외적인 상황들과, 그때 어떤 원칙으로 판단해야 하는지를 다룬다. 다루는 상황은 다음과 같다.

- 태그 포맷이 레포 히스토리 안에서 섞여 있는 경우
- 태그와 파일에 적힌 버전이 서로 다른 경우
- 첫 릴리스의 시작 버전을 정해야 하는 경우
- 과거 릴리스 노트가 아예 없던 레포를 소급 정리하는 경우
- 잘못 나간 버전을 회수해야 하는 경우
- 릴리스가 중간에 멈춘 채로 남아 있는 경우
- init이 만들려는 파일 자리에 이미 다른 파일이 있는 경우

## 혼재된 태그 포맷

`v1.2`, `1.2.0`, `release-1.2`처럼 서로 다른 태그 포맷이 레포 히스토리에 섞여 있는 경우가 있다.

과거에 이미 찍힌 태그는 손대지 않는다 — 지금 와서 다시 태깅하거나 하나로 통일하려 하지 않는다. 대신 앞으로 어떤 포맷을 표준으로 쓸지만 사용자에게 질문한다.

anchor(마지막 릴리스 지점) 계산도 단순히 시간상 가장 최근 태그를 고르는 것이 아니라, 새로 정한 표준 포맷과 일치하는 태그를 우선으로 삼는다.

worked example: 히스토리에 `v1.2`, `v1.3`, `1.4.0`, `1.5.0`이 순서대로 섞여 있고 이번에 `1.4.0`부터의 포맷(v prefix 없음)을 표준으로 확정했다고 하자. 이때 anchor는 시간순으로 가장 최근인 태그가 아니라, 그 표준과 일치하는 가장 최근 태그인 `1.5.0`을 기준으로 잡는다. 그래야 과거의 임시 포맷 태그 때문에 범위 계산이 엉키지 않는다.

## 태그·파일 버전 불일치

빌드 파일에 적힌 버전과 마지막 태그가 가리키는 버전이 다를 수 있다.

이럴 때는 어느 쪽을 신뢰할지 사용자에게 질문하되, 기본값은 **파일이 SSOT**라는 원칙을 따른다 — 태그는 파생물이라는 이 플러그인의 기본 전제와 일관된다.

불일치가 생기는 흔한 원인 후보:

- 수동으로 태그를 찍은 경우
- 이전 릴리스가 도중에 중단된 경우

원인을 좁힐 때는 마지막 태그가 가리키는 커밋과 현재 파일 버전이 반영된 커밋을 비교해보는 것이 도움이 된다. 둘이 같은 커밋 근방이면 태그를 나중에 수동으로 옮긴 것에 가깝고, 파일 버전 쪽이 태그보다 훨씬 앞서 있다면 아래에서 다룰 "중단 상태"일 가능성이 크다.

## 첫 릴리스: 0.1.0 vs 1.0.0

기존 버전이 아예 없는 첫 릴리스라면 0.1.0으로 시작할지 1.0.0으로 시작할지를 질문한다.

판단 기준은 SemVer에서 1.0.0이 갖는 의미, 즉 "공개 API를 안정적으로 지키겠다는 약속"을 이미 할 수 있는 상태인지다.

- 아직 API가 크게 바뀔 여지가 많다면 → 0.1.0이 정직한 선택
- 이미 안정성을 약속할 준비가 됐다면 → 1.0.0으로 시작해도 된다

## CHANGELOG backfill

이미 태그가 쌓여 있는 기존 레포에서, 그동안 기록되지 않았던 과거 릴리스 노트를 소급 작성하는 기능이다.

태그 구간(태그 A와 태그 B 사이)별로 그 사이의 커밋을 모아 각 릴리스의 노트를 사후에 채워 넣는 방식이다.

worked example: 태그가 `v1.0.0`부터 `v1.5.0`까지 다섯 개 쌓여 있는데 CHANGELOG가 전혀 없었다면, backfill은 `v1.0.0..v1.1.0`, `v1.1.0..v1.2.0`처럼 태그 구간을 순서대로 훑어 각 구간의 커밋으로 그 릴리스의 노트를 하나씩 사후 작성한다.

**이 기능은 M3에서 지원되며, M1에는 포함되지 않는다.**

## 버전 재사용 금지

한 번 발행된 버전은 절대로 다시 태깅하지 않는다. 이유는 두 가지다.

- 소비자가 이미 그 버전을 내려받아 캐시해뒀을 수 있다 — 나중에 같은 버전 번호가 다른 내용을 가리키게 되면 소비자마다 서로 다른 아티팩트를 보게 되는 모순이 생긴다.
- 버전 번호는 한 번 발행되면 그 내용이 바뀌지 않는다는 암묵적 불변성 계약으로 취급된다 — 패키지 레지스트리, 빌드 캐시, lockfile 모두 이 전제를 깔고 동작하므로 이를 깨면 생태계 전반의 캐시 신뢰가 무너진다.

같은 원리가 Docker 이미지 태그에도 적용된다. 채널 태그(`dev`, `latest`)는 원래 가변적이지만, 한 번 발행된 릴리스용 태그(`v1.3.0`처럼 특정 버전을 가리키는 태그)를 다른 이미지로 다시 push하는 것은 이 원칙을 어기는 것과 같다. 진짜 불변 식별자는 태그가 아니라 이미지 다이제스트라는 점도 함께 기억해둘 만하다(웹 앱 dev 채널의 불변 sha 병기와 같은 맥락 — `prerelease-and-dev-channel.md` 참고).

잘못된 버전이 이미 나갔다면 회수 절차는 다음과 같다.

- 재태깅하지 않고 **다음 patch 버전으로 덮는다** — 이것이 기본 원칙이다.
- npm 생태계라면 `npm deprecate <패키지>@<버전> "메시지"`로 해당 버전에 경고 문구를 건다.
- PyPI라면 해당 릴리스를 yank 처리해 신규 설치 대상에서 제외한다(완전히 삭제하지는 않고, 이미 그 버전을 명시적으로 고정한 소비자에게는 계속 남아 있다).

## 중단 상태 복구

파일 버전이 마지막 태그보다 앞서 있는데 그 버전에 대한 태그가 아직 없다면, 이전 릴리스가 어딘가에서 중단됐다는 뜻이다. 이때 두 가지 선택지를 제시한다.

- **resume**: 중단된 지점 이후 남은 단계(노트 작성, 태깅, push 등)를 이어서 진행한다.
- **rollback**: 파일에 반영된 버전을 이전 상태로 되돌린다. 단, 이미 원격에 push된 커밋·태그는 되돌리지 않는다 — 되돌릴 수 있는 대상은 아직 push되지 않은 로컬 상태로 한정된다. 이는 버전 재사용 금지 원칙과도 맞닿아 있다 — 이미 밖으로 나간 것은 없앨 수 없고 덮을 수만 있다.

resume과 rollback 중 무엇을 고를지는 이미 얼마나 진행됐는지에 달려 있다.

- 아직 로컬에서 파일 버전만 바꾼 상태라면 rollback이 간단하고 안전하다.
- 노트 작성까지 끝내고 검토까지 받은 상태라면 그 작업을 버리기보다 resume으로 이어가는 편이 낫다.

## init 충돌

init이 생성하려는 목적지 경로에 generated 마커 없는 파일이 이미 있으면 경고를 띄우고 덮어쓸지 확인을 받는다 — 이번이 이 레포의 첫 init이더라도 예외는 아니다.

마커가 없다는 것은 그 파일이 superrelease가 만든 게 아니라 누군가 손으로 작성했다는 뜻으로 간주하기 때문이다. 예를 들어 이 레포에 원래부터 손으로 작성한 `.claude/skills/release/SKILL.md`가 있었다면(superrelease 도입 이전부터 있던 파일), 그 파일에는 generated 마커가 없으므로 init은 이를 곧바로 덮어쓰지 않고 경고와 함께 확인을 받는다.

다만 템플릿으로 지정된 파일은 예외적으로 손편집이 허용되는 유일한 영역이다. 마커가 제거되어 있으면 이를 충돌이 아니라 "의도된 손편집"으로 보고 경고 없이 그대로 보존하고 건너뛴다는 점이 다르다.
