# bump 판단 모델

버전을 얼마나 올릴지 자동으로 판단하는 도구들은 서로 접근 방식이 다르다.

superrelease의 위치를 이해하려면 먼저 널리 쓰이는 세 모델을 비교해볼 필요가 있다.

## 3모델 비교

| 모델 | 동작 방식 | 사람 개입 시점 |
|---|---|---|
| semantic-release | 커밋을 분석해 다음 버전을 계산하고 그대로 배포까지 진행 | 완전 자동 — 별도 확인 단계 없음 |
| release-please | 병합될 때마다 "릴리스 PR"을 누적 갱신해두고, 그 PR이 머지되는 순간이 곧 릴리스 | 릴리스 PR을 머지하는 행위 자체가 승인 |
| changesets | 기여자가 PR에 변경 의도를 담은 파일(changeset)을 커밋해두고, 릴리스 시 이를 모아 버전과 노트를 계산 | 변경 시점마다 사람이 의도를 직접 기술 |

같은 상황을 세 모델에 각각 대입해보면 차이가 분명해진다. 마지막 릴리스 이후 `feat` 커밋 2건과 `fix` 커밋 1건이 쌓였다고 하자.

- **semantic-release**: push 즉시 이를 분석해 MINOR 버전을 계산하고 그대로 배포까지 마친다.
- **release-please**: 이 커밋들이 쌓일 때마다 "다음 버전은 MINOR"라고 표시된 릴리스 PR의 내용이 계속 갱신되다가, 누군가 그 PR을 머지하는 순간 릴리스가 이루어진다.
- **changesets**: 애초에 이 세 커밋 각각의 PR에 "이건 minor", "이건 patch" 같은 changeset 파일이 함께 커밋되어 있어야 하고, 릴리스 시점에 이 파일들을 모아 최종 bump 수준을 계산한다.

changesets는 changeset 파일 하나가 모노레포 안 여러 패키지에 대해 각각 다른 bump 수준을 동시에 지정할 수 있어, 모노레포에서 특히 강점을 보인다.

superrelease의 `bump.mode: auto-confirm`은 이 셋 사이의 절충이다.

- 커밋·PR 분석으로 bump를 자동 산출하는 점은 semantic-release와 같다.
- 그 결과를 곧바로 배포에 반영하지 않고 항상 근거와 함께 제시한 뒤 확인을 받는다.

한 줄로 요약하면 "분석은 자동, 결정은 확인"이다.

## Conventional Commits 매핑

- `feat` → MINOR
- `fix` → PATCH
- `BREAKING CHANGE:` 푸터, 또는 타입/스코프 뒤에 붙는 `!`(예: `feat!:`) → MAJOR
- 그 외 타입(`docs`, `chore`, `refactor`, `style`, `test`, `build`, `ci` 등) → bump 없음

config `bump.sources`에 `conventional-commits`가 포함돼 있을 때 이 매핑이 1차 분석 규칙으로 쓰인다.

타입 뒤 괄호로 붙는 스코프(예: `feat(auth):`)는 bump 판단 자체에는 관여하지 않고 분류·가독성 용도로만 쓰인다.

이 매핑은 Conventional Commits 스펙이 명시적으로 못박은 부분(`feat`/`fix`/`BREAKING CHANGE`)과, 스펙이 권장만 하고 판단은 도구에 맡긴 부분(그 외 타입)으로 나뉜다는 점도 참고할 만하다 — `perf`처럼 릴리스 도구에 따라 patch로 취급하기도 하는 타입이 있지만, superrelease는 스펙이 명시한 세 가지 규칙만 기본으로 삼는다.

## squash 레포에서 PR 메타데이터가 1차 소스인 이유

squash merge를 쓰는 레포는 PR 하나가 브랜치에는 커밋 하나로 남는다.

이 커밋의 제목은 보통 PR 제목 뒤에 `(#N)`이 붙는 형태(예: `feat: 소셜 로그인 추가 (#42)`)다.

문제는 PR 안에서 오갔던 개별 커밋들(작업 중 남긴 `wip`, `fix typo` 같은 커밋)의 메시지는 이 과정에서 브랜치 히스토리에서 사실상 사라진다는 점이다 — squash 커밋 하나만 남고 그 세부 이력은 더 이상 `git log`로 보이지 않는다.

그래서 squash 레포에서는 개별 커밋 메시지보다 **PR 메타데이터(제목·본문·라벨·연결된 이슈)가 1차 소스**가 된다.

커밋에서 PR로 역참조하는 방법은 두 가지다.

1. 커밋 제목의 `(#N)` 패턴을 파싱해서 PR 번호를 얻는다.
2. 제목에 그 패턴이 없거나 확실히 확인하고 싶으면 `gh api repos/{owner}/{repo}/commits/{sha}/pulls`로 해당 커밋과 연관된 PR을 API로 직접 조회한다.

반대로 squash를 쓰지 않는 레포(merge commit이나 rebase 기반)라면 개별 커밋이 브랜치 히스토리에 그대로 남으므로 커밋 메시지 자체를 1차 소스로 써도 무리가 없다. PR 메타데이터로 우회할 필요가 squash 레포보다 적다.

## diff는 검증용

커밋·PR 메타데이터만으로 bump 수준이나 breaking 여부가 애매할 때만 실제 diff를 열어본다.

diff 전체를 매번 읽는 것은 토큰 비용이 크므로, 먼저 변경된 파일 목록만 훑어보고 그중 의심되는 파일 몇 개만 골라 diff를 확인하는 순서를 지킨다 — 전수조사가 아니라 애매할 때의 보조 확인 수단이라는 점이 핵심이다.

예를 들어 PR 하나가 `package-lock.json`처럼 수천 줄이 바뀐 파일을 포함하고 있다면, 그 diff를 통째로 읽는 것은 낭비다.

자동 생성된 lockfile류는 건너뛰고, 실제 로직이 담긴 소스 파일 몇 개만 의심 대상으로 골라 diff를 열어보는 식으로 접근한다.

## 라이브러리 API 호환성 도구

메시지 기반 판단(커밋 타입, PR 라벨)은 결국 사람이 "이건 breaking이야"라고 스스로 분류한 결과에 의존한다.

라이브러리라면 여기서 한 단계 더 나아가, 실제 공개 API의 바이너리·시그니처 diff로 breaking 여부를 기계적으로 판정하는 도구를 쓸 수 있다.

- **Kotlin binary-compatibility-validator**: 공개 API 시그니처를 파일로 덤프해두고 버전 간 차이를 비교해 바이너리 비호환 변경을 잡아낸다. 보통 Gradle 플러그인으로 붙어 `apiDump`/`apiCheck` 같은 태스크를 제공한다.
- **japicmp**: 두 Java 아티팩트(JAR)를 바이트코드 수준에서 비교해 소스·바이너리 호환성 변화를 분류한다. Maven·Gradle 플러그인 형태로 빌드 단계에 통합되는 경우가 많다.

이 도구들의 핵심은 "커밋 메시지가 아니라 실제 바이너리 diff로" breaking을 판정한다는 점이다. 사람이 커밋 메시지에 `!`를 붙이는 걸 깜빡해도 실제 API가 깨졌다면 놓치지 않는다.

이런 도구가 "바이너리 비호환"이라고 판정하면, 설령 커밋 메시지에 breaking 표시가 없었더라도 MAJOR로 재분류하는 근거가 된다.

config의 `bump.compatCheck`에는 이런 도구의 식별자(예: `kotlin-bcv`, `japicmp`)를 기록해둔다.

**M1에서는 이 값을 기록하고 관련 도구 사용법을 안내하는 역할까지만 하며, 실제 검사를 자동 실행하지는 않는다.**

## config 필드 3종의 역할 분담

bump 판단과 관련된 config 필드는 세 가지로 나뉘며, 서로 대체 관계가 아니라 단계별 보완 관계다.

- `bump.sources`: 배열로, 커밋 기반 분석의 우선순위를 나열한다(예: `["conventional-commits", "pr-metadata"]` — squash 레포라면 `pr-metadata`가 사실상 1차가 된다).
- `bump.fallback`: 위 소스만으로 판단이 애매할 때만 넘어가는 보조 수단으로, M1에서는 `diff`(실제 변경 내용 확인)가 이 자리를 채운다.
- `bump.compatCheck`: 라이브러리에서만 의미가 있는 별도 필드로, API 호환성 도구 식별자를 기록해 메시지 기반 판단을 바이너리 diff로 보강한다.

세 필드를 매번 전부 실행하는 것이 아니라, `sources`가 명확한 답을 주면 거기서 멈추고, 애매함이 남았을 때만 `fallback`으로, 라이브러리라면 `compatCheck` 도구 결과로 보강하는 순서다.
