# 버전 체계 선택 가이드

이 문서는 init이 스코프별 버전 체계를 제안하거나 사용자의 선택을 검토할 때 참고하는 배경 지식이다.

프로젝트 성격에 따라 적합한 체계가 갈리므로, 아래 요약표로 방향을 먼저 잡고 각 절에서 세부 규칙을 확인한다.

| 체계 | 가장 잘 맞는 대상 | 핵심 아이디어 |
|---|---|---|
| SemVer | 라이브러리 | 공개 API를 소비하는 곳이 있으면 사실상 강제되는 표준 |
| CalVer | release train, 주기 배포 서비스 | 버전 자체가 "언제" 릴리스됐는지를 말해준다 |
| HeadVer | 앱, 서비스 | 사람이 정할 부분과 자동 계산할 부분을 분리해 버전 논쟁을 없앤다 |
| Sequential | 내부 도구 | 그냥 1씩 늘어나는 정수, 의미 부여 없음 |

## SemVer

SemVer(Semantic Versioning)는 `MAJOR.MINOR.PATCH` 세 자리로 구성된다.

- MAJOR — 기존 공개 API와 호환되지 않는 변경
- MINOR — 하위 호환을 유지하면서 기능을 추가한 변경
- PATCH — 하위 호환을 유지하는 버그 수정

이 규칙 덕분에 소비자는 버전 번호만 보고도 업그레이드 시 무엇을 조심해야 하는지 가늠할 수 있다. 그래서 다른 패키지가 의존하는 라이브러리라면 SemVer 채택이 사실상 강제된다고 본다.

버전 문자열에는 두 가지 부가 요소가 붙을 수 있다.

- **pre-release**: `-alpha.1`, `-beta.2`, `-rc.1`처럼 하이픈 뒤에 붙는 식별자. 정식 버전보다 낮은 우선순위로 취급되며, 아직 정식 릴리스 전 단계임을 나타낸다.
- **build metadata**: `+build.5`, `+20260709`처럼 plus 기호 뒤에 붙는 식별자. 버전 우선순위 비교에는 관여하지 않는 순수 부가 정보다.

pre-release 식별자끼리도 순서가 있다. 점(`.`)으로 구분된 필드를 왼쪽부터 비교하며, 숫자로만 된 필드는 산술 비교로, 문자가 섞인 필드는 아스키 사전순으로 비교한다.

그 결과 정렬 순서는 다음과 같다: `1.0.0-alpha` < `1.0.0-alpha.1` < `1.0.0-beta` < `1.0.0-rc.1` < `1.0.0`. 정식 버전이 가장 나중이라는 원칙과도 일치한다.

**0.x 관례**: SemVer 스펙은 `0.y.z`를 초기 개발 단계로 규정한다. 이 구간에서는 언제든 무엇이든 바뀔 수 있고 공개 API가 안정적이라고 간주하지 않는다.

이 때문에 많은 프로젝트가 관례적으로 0.x 구간에서는 breaking change를 MINOR 증가로, 일반 기능 추가를 PATCH 증가로 다룬다. MAJOR는 0에 묶어두고 1.0.0으로의 도약을 의도적으로 미루는 방식이다. superrelease도 이 관례를 bump 제안에 그대로 반영한다.

**1.0.0의 의미**: 1.0.0부터는 공개 API가 확정된다는 뜻이다. 이후 버전 번호가 어떻게 올라가는지는 전적으로 그 공개 API가 어떻게 바뀌는지에 달려 있다.

즉 1.0.0을 붙인다는 것은 "이 API를 안정적으로 지키겠다"는 약속이며, 아직 그 약속을 할 준비가 안 됐다면 0.x에 머무는 편이 정직하다.

다음 버전 계산은 사람이나 LLM이 암산하지 않고 스크립트로 산출한다. 현재 버전이 `1.2.3`일 때의 예시는 다음과 같다.

- `next-version.py --bump patch` → `1.2.4`
- `next-version.py --bump minor` → `1.3.0`
- `next-version.py --bump major` → `2.0.0`
- `next-version.py --release` (현재 버전이 `1.3.0-SNAPSHOT`일 때) → `1.3.0`

참고: [semver.org](https://semver.org/), [github.com/semver/semver](https://github.com/semver/semver)

## CalVer

CalVer(Calendar Versioning)는 버전의 일부 혹은 전부를 달력에서 가져온다. 흔히 쓰는 패턴 어휘는 다음과 같다.

- `YYYY` — 전체 연도 (2026)
- `YY` — 두 자리 연도 (26)
- `0M` — 0으로 채운 두 자리 월 (01~12)
- `MM` — 0을 채우지 않은 월 (1~12)
- `0D` — 0으로 채운 두 자리 일 (01~31)
- `DD` — 0을 채우지 않은 일 (1~31)
- `MICRO` — 달력과 무관하게 같은 기간 안에서 그냥 증가하는 카운터

CalVer 표준에는 이 밖에도 `WW`(연중 주차)·`0Y`(0으로 채운 연도) 같은 토큰이 정의돼 있다. 다만 **superrelease의 `next-version.py`가 실제로 계산하는 어휘는 `YYYY`·`YY`·`0M`·`MM`·`0D`·`DD`·`MICRO` 7종뿐이다.** 패턴에 그 외 토큰(예: `WW`)을 쓰면 주차 등으로 산술되지 않고 리터럴 문자로 그대로 남으므로, config의 `scheme.pattern`에는 이 7종만 사용한다.

대표 사례 두 가지를 보면 패턴의 쓰임새가 분명해진다.

- **Ubuntu**: `YY.0M` 패턴 (20.04, 22.04, 24.04 — 매년 4월·10월 릴리스)
- **pip**: `YY.N` 패턴 (20.0, 20.1, 20.2 …). pip은 2020년 20.0 버전부터 이 체계로 전환한다고 공식적으로 밝힌 바 있다. 연도 뒤에 그 해의 몇 번째 릴리스인지를 나타내는 MICRO 카운터가 붙는 형태다.

CalVer는 release train과 궁합이 좋다. 정기적으로 묶어서 내보내는 릴리스라면 "언제 나온 train인가"가 버전 자체에 드러나는 편이 자연스럽기 때문이다.

Spring Cloud가 대표적인 사례다 — 개별 모듈은 SemVer를 쓰면서 루트 release train 이름만 `2020.0.x` 같은 CalVer 스타일로 옮겨갔다(모노레포에서의 이중 체계 조합은 `monorepo.md` 참고).

다만 CalVer가 항상 정답은 아니다. 라이브러리처럼 소비자가 호환성 신호를 필요로 하는 대상에는 날짜 자체가 아무 의미도 없으므로 SemVer가 훨씬 유용하다.

날짜·주차 산술은 사람이 암산하거나 LLM이 즉석에서 계산하면 안 되고 반드시 스크립트를 거쳐야 한다.

**CalVer의 다음 버전 계산은 `next-version.py --scheme calver --pattern <PATTERN> [--today YYYY-MM-DD]`가 수행한다** — 같은 기간이면 MICRO를 증가시키고, 기간이 바뀌면 0으로 리셋한다(MICRO가 없는 패턴은 날짜 렌더만 반환). config 모드(`--scope`)에서는 scheme.pattern을 자동 사용한다.

참고: [calver.org](https://calver.org/), [github.com/mahmoud/calver](https://github.com/mahmoud/calver)

## HeadVer

HeadVer는 LINE(현 LY Corporation)이 제안한 체계로, `{head}.{yearweek}.{build}` 세 자리로 구성된다.

- **head**: 사람이 직접 정하는 값. 제품 방향에 따른 결정(예: 큰 개편)이며 자동으로는 바뀌지 않는다.
- **yearweek**: 두 자리 연도와 ISO 주차를 합쳐 자동으로 계산되는 값(예: 2026년 28주차 → 2628).
- **build**: 빌드할 때마다 자동으로 증가하는 카운터.

worked example로 감을 잡아보면 이렇다.

- head를 3으로 정해둔 상태에서 2026년 28주차에 12번째 빌드를 만들면 버전은 `3.2628.12`가 된다.
- 같은 주에 빌드를 한 번 더 하면 `3.2628.13`이 된다.
- 다음 주로 넘어가면 yearweek만 `2629`로 바뀌고 build는 이전 값에서 이어서 증가한다 (`3.2629.14`) — 이 프로젝트가 채택하는 계산 규칙 기준으로 build는 주가 바뀌어도 초기화되지 않는다.

설계 의도는 "버전 논쟁을 없앤다"는 것이다. SemVer처럼 "이게 breaking인가 아닌가"를 매번 논쟁하는 대신, head는 사람이 가볍게 정하고 나머지 두 자리는 기계적으로 채워 넣는다.

이 구조는 공개 API 호환성 계약이 뚜렷하지 않은 앱·서비스에 잘 맞는다. 앱은 라이브러리와 달리 "누군가 이 API 시그니처에 코드로 의존한다"는 전제가 약하기 때문에, breaking/non-breaking을 가르는 SemVer식 판단 자체가 잘 들어맞지 않는 경우가 많다.

**다음 버전 계산(yearweek 산출, build 증가)은 `next-version.py --scheme headver --head <N> [--today YYYY-MM-DD]`가 수행한다** — yearweek는 ISO 연도 2자리+ISO 주차 2자리(1월 초는 전년도 주차일 수 있음), build는 현재 버전의 3번째 필드+1(리셋 없음)이다. config에는 head 번호를 `scheme.pattern`에 기록하며 config 모드에서 자동 사용된다.

참고: [github.com/line/headver](https://github.com/line/headver), [LY 기술블로그 소개글](https://techblog.lycorp.co.jp/ko/headver-new-versioning-system-for-product-teams)

## Sequential

가장 단순한 체계로, `1`, `2`, `3`처럼 정수를 하나씩 늘린다.

날짜도 호환성도 나타내지 않으므로 CI 빌드 번호(`build #482`)나 사내 배포 카운터처럼 외부 소비자가 없는 내부 도구에 적합하다.

공개 라이브러리나 사용자에게 노출되는 앱에는 권장 범위가 좁다. 버전 번호 자체가 아무 의미도 전달하지 못해서, 소비자가 업그레이드 시 무엇을 조심해야 하는지 전혀 짐작할 수 없기 때문이다.

## init 기본값 규칙

- **라이브러리** → SemVer가 기본값이며 사실상 강제에 가깝다. 다른 체계를 선택하면 그 선택이 소비자에게 어떤 불편을 주는지 경고를 보여주고 재확인을 받는다.
- **앱/서비스** → SemVer가 기본값이되, CalVer나 HeadVer도 대안으로 함께 제안한다.
- **release train 루트** → CalVer를 기본값으로 제안한다.

라이브러리에 SemVer를 사실상 강제하는 이유는 도덕적 규범이 아니라 실용적인 이유다. npm의 `^1.2.3`, Gradle/Maven의 버전 범위 지정처럼 패키지 매니저의 의존성 해석 규칙 자체가 SemVer 순서를 전제로 설계돼 있다.

CalVer나 Sequential 버전을 라이브러리에 쓰면 이런 범위 연산자가 기대한 대로 동작하지 않아 의존성 해석이 어긋날 수 있다.

**SemVer·CalVer·HeadVer 세 체계 모두 지원된다** — 날짜·주차·카운터 산술은 전부 `next-version.py`가 수행하며(LLM 산술 금지), `--today` 주입으로 결정론적으로 테스트된다.

sequential(단순 증가)은 아직 구현되어 있지 않다 — 질문 단계에서 후보로 떠오르면 "후속 버전에서 지원 예정"으로 안내한다.
